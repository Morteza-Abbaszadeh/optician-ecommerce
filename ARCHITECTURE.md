# 📚 مستندات جامع معماری و توسعه پروژه فروشگاه عینک

## ۱. نمای کلی سیستم (System Overview)

این پروژه یک پلتفرم فروشگاهی حرفه‌ای با معماری **Modular Monolith** است. سیستم به دو بخش کاملاً مجزا (Decoupled) تقسیم شده است:

* **فرانت‌اند (Frontend):** توسعه‌یافته با `Next.js` (App Router) و `TypeScript`. وظیفه آن مدیریت رابط کاربری، Stateها و تعامل با کاربر است.
* **بک‌اند (Backend):** توسعه‌یافته با `Python` و فریم‌ورک `FastAPI`. وظیفه آن مدیریت منطق تجاری (Business Logic)، ارتباط با دیتابیس (`PostgreSQL`) و ارائه API است.
* **زیرساخت (Infrastructure):** کل پروژه در محیط `Docker` کانتینرایز شده است تا در هر محیطی (توسعه یا پروداکشن) یکسان اجرا شود.

## ۲. کالبدشکافی بک‌اند (FastAPI Backend)

بک‌اند ما بر اساس ساختار **ماژولار (Domain-Driven)** چیده شده است. به جای اینکه تمام فایل‌ها روی هم تلنبار شوند، هر بخش از کسب‌وکار (مثل کاربران، محصولات، سفارشات) پوشه مخصوص به خود را دارد.

### ساختار پوشه‌ها:

```text
backend/app/
├── core/         # تنظیمات اصلی سیستم (اتصال دیتابیس، امنیت، متغیرهای محیطی)
├── shared/       # کدهای مشترک که در همه جا استفاده می‌شوند (مثل توابع کمکی)
└── modules/      # قلب تپنده پروژه؛ شامل ماژول‌های مستقل:
    ├── users/    # مدیریت کاربران و احراز هویت
    ├── products/ # مدیریت کالاها، تنوع‌ها و موجودی
    └── orders/   # مدیریت سبد خرید و صدور فاکتور

```

### آناتومی هر ماژول (چرا این همه فایل داریم؟)

در هر پوشه ماژول (مثلاً `orders`) فایل‌های زیر وجود دارند که هر کدام وظیفه بسیار مشخصی دارند:

1. **`models.py` (نقشه دیتابیس):**
* **کاربرد:** این فایل با استفاده از `SQLAlchemy` نوشته می‌شود. فقط و فقط برای ساخت جداول و ستون‌ها در دیتابیس `PostgreSQL` است.
* **مثال:** تعیین اینکه `total_amount` از نوع Float است و `id` از نوع UUID.


2. **`schemas.py` (دروازه‌بان و اعتبارسنج):**
* **کاربرد:** این فایل با `Pydantic` نوشته می‌شود. وظیفه آن بررسی اطلاعاتی است که از سمت فرانت‌اند می‌آید (Input) و اطلاعاتی که به سمت فرانت‌اند می‌رود (Output).
* **نکته کلیدی:** اگر فرانت‌اند فیلد اشتباهی بفرستد، این فایل قبل از رسیدن دیتا به دیتابیس، ارور `422` یا `500` می‌دهد.


3. **`repository.py` (مغز متفکر / منطق تجاری):**
* **کاربرد:** تمام منطق پیچیده (Business Logic) در اینجا نوشته می‌شود. بررسی موجودی انبار، کسر از انبار، محاسبه قیمت‌ها و صدور دستور `INSERT` به دیتابیس.
* **چرا؟** برای اینکه کدهای API (روتر) شلوغ نشوند و بتوانیم این توابع را در جاهای دیگر هم استفاده کنیم.


4. **`router.py` (پریزه برق / API Endpoints):**
* **کاربرد:** آدرس‌های URL مثل `POST /api/v1/orders/` در اینجا تعریف می‌شوند. روتر فقط دیتا را از فرانت‌اند می‌گیرد، به `repository` می‌دهد و جواب را برمی‌گرداند (فقط نقش رابط را دارد).

## ۳. کالبدشکافی فرانت‌اند (Next.js Frontend)

فرانت‌اند با تمرکز بر پرفورمنس و سئو (SEO) طراحی شده است.

### ساختار پوشه‌ها:

```text
frontend/src/
├── app/          # صفحات سایت (Router). مثلاً app/cart/page.tsx
├── components/   # قطعات رابط کاربری (دکمه‌ها، فرم‌ها، هدر) که در صفحات تکرار می‌شوند
├── store/        # حافظه کوتاه‌مدت سایت (Zustand)
└── services/     # واسط ارتباط با بک‌اند (Axios)

```

### فایل‌های کلیدی و وظایف آن‌ها:

* **`store/` (مثلاً `useCartStore.ts`):** حافظه سراسری مرورگر. مثلاً وقتی محصولی به سبد خرید اضافه می‌شود، در اینجا ذخیره می‌شود تا کاربر با رفتن به صفحه دیگر، سبد خریدش پاک نشود (مدیریت State با Zustand).
* **`services/` (مثلاً `orderService.ts`):** اینجا دقیقاً جایی است که فرانت‌اند با بک‌اند تماس می‌گیرد. کدهای `axios.post` در اینجا هستند تا اگر آدرس بک‌اند عوض شد، فقط یک فایل را تغییر دهیم، نه کل صفحات را.

## ۴. جریان داده (Data Flow) - وقتی دکمه‌ای کلیک می‌شود چه می‌شود؟

بیایید مسیر دکمه **"تکمیل خرید"** را قدم به قدم ردیابی کنیم:

1. **کاربر کلیک می‌کند:** در فرانت‌اند (`cart/page.tsx`)، تابع `handleCheckout` اجرا می‌شود.
2. **ارسال درخواست:** تابع `orderService.createOrder()` دیتا را به شکل JSON بسته‌بندی کرده و به مسیر `/orders/` بک‌اند می‌فرستد.
3. **دروازه‌بان می‌بیند (schemas.py):** بک‌اند دیتا را می‌گیرد و با `OrderCreate` در فایل Schema چک می‌کند. اگر درست بود، اجازه ورود می‌دهد.
4. **مغز تصمیم می‌گیرد (repository.py):** سیستم چک می‌کند که آیا کالا موجودی دارد؟ اگر نداشت ارور `ValueError` (کد 400) صادر می‌کند. اگر داشت، قیمت‌ها را جمع می‌زند و فاکتور را می‌سازد.
5. **ذخیره در دیتابیس (models.py):** فاکتور با دستورات `SQLAlchemy` در جداول PostgreSQL ذخیره می‌شود.
6. **بسته‌بندی جواب (schemas.py):** بک‌اند آیدی فاکتور و وضعیت را طبق قالب `OrderResponse` به JSON تبدیل کرده و می‌فرستد. (اینجا بود که خطای تبدیل UUID به String را حل کردیم).
7. **نمایش موفقیت:** فرانت‌اند جواب را می‌گیرد، سبد خرید (`Zustand`) را خالی می‌کند و کاربر را به مسیر `/cart/success` می‌فرستد.

## ۵. راهنمای ابزارها و دیتابیس (Alembic & Docker)

### مدیریت دیتابیس با Alembic (Migrations):

وقتی تغییری در `models.py` می‌دهید (مثلاً یک ستون جدید اضافه می‌کنید)، دیتابیس خودبه‌خود متوجه نمی‌شود. باید مراحل زیر را طی کنید:

1. **ثبت تغییرات (ساخت فایل اسکریپت):**
```bash
docker exec -it optician_backend alembic revision --autogenerate -m "توضیح تغییر"

```


2. **اجرای تغییرات روی دیتابیس:**
```bash
docker exec -it optician_backend alembic upgrade head

```



### دستورات حیاتی داکر (Cheat Sheet):

* **دیدن لاگ‌های بک‌اند (برای پیدا کردن ارور ۵۰۰):**
```bash
docker logs optician_backend --tail 50 -f

```


* **ری‌استارت کردن بک‌اند بعد از تغییر کد:**
```bash
docker restart optician_backend

```


* **ورود مستقیم به دیتابیس (برای چک کردن جداول):**
```bash
docker exec -it optician_db psql -U optician_user -d optician_db

```

## ۶. درس‌های آموخته شده (Lessons Learned)

اگر در آینده به باگ خوردید، ابتدا این ۳ مورد را چک کنید:

1. **ارور ۵۰۰ (`Internal Server Error`):** هرگز از روی فرانت‌اند حدس نزنید. بلافاصله لاگ داکر بک‌اند را چک کنید. معمولاً مشکل از تطابق نداشتن `Schema` (مثلاً UUID با String) است.
2. **ارور شبکه (`CORS` یا `Network Error`):** اگر سرور روشن است اما ارور CORS می‌گیرید، به احتمال ۹۹٪ مشکل از قرار ندادن **اسلش آخر (/)** در آدرس‌های فرانت‌اند (`axios`) است. FastAPI به اسلش آخر حساس است.
3. **ارور ۴۰۰ (`Bad Request`):** این یعنی کدهای شما درست کار می‌کند، اما دیتای فرانت‌اند از نظر منطقی غلط است (مثلاً خرید کالایی که موجودی صفر دارد). همیشه متن ارور (`detail`) را از بک‌اند بخوانید و به کاربر نشان دهید.
